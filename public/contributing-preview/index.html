Before you write
01

Where work happens

main is the protected release line — whatever version is shipping next. The backlog lives on next. The community works in forks. A branch is not the shipped surface — so nothing that isn’t in a release epic ever reaches the build.

Hard stop

Never push to main. It changes only through reviewed, milestone-linked PRs. Direct pushes, force-pushes, and un-reviewed merges are blocked by ruleset — don’t try to route around them.

Pick your base by what the work is
  • →Fixing a bug, or building a capability already locked for the current release — branch from main, PR into main. This is in-scope release work.
  • →Anything beyond the current release, or not yet scoped — branch from next, PR into next. It rides the backlog line and never touches the shipped code.
  • →Not a core maintainer — fork, then PR from your fork into the matching base above. You don’t need write access to contribute.

main is merged down into next regularly so the backlog never drifts from the release. Never rebase a shared branch — main and next are merged, not rebased.

02

Check scope before you write code

The locked scope lives in the release’s scope-of-record file (currently docs/architecture/v1-scope.md). It is the source of record for what the current release contains — not an issue thread, not a Discord message, not this page.

Hard stop

Do not open a feature PR for a capability that isn’t locked. If it isn’t in v1-scope.md — or the file says scope isn’t locked yet — file a capability proposal issue and wait for triage. A PR with no locked scope item behind it will be closed.

Proceeds without a proposal
  • ✓Bug fixes
  • ✓Refactors & cleanup that don’t change behavior
  • ✓Work on a capability already locked in v1-scope.md
Needs a proposal + triage first
  • •Any new user-facing capability
  • •A new library type (see Article 03)
  • •Anything that adds surface to the client↔server API
03

Proposing new work

Two parent templates, one chooser, in Silo-Server/.github/ISSUE_TEMPLATE. Pick by what kind of thing you’re proposing — the gates differ. Each creates a typed parent issue — a Feature or a Library Type; the per-surface work then arrives as Task sub-issues, and defects as Bugs.

feature.yml

A generic user-facing capability.

  • •One proposal per capability — not per PR, not per endpoint.
  • •User value: why this release rather than a later one.
  • •Per-client impact (android / apple / web) — “none” is a valid answer.

library-type.yml

A new kind of library — carries an extra gate.

  • •Everything feature.yml asks, plus:
  • •The validating first-party metadata plugin(s).
  • •Marker applicability (Movie/Series only) & provider priority.
Rule

A capability cannot be locked without its API surface filled in — endpoints, methods, payloads. A library type cannot be filed without naming the first-party plugin that validates it. The template enforces the contract so the boundary is real, not aspirational.

How work is tracked
04

How your work is tracked

Every locked capability is tracked as one parent issue — a Feature or Library Type — with one Task sub-issue per surface it ships on. The parent holds the acceptance criteria but carries no surface of its own; it’s the umbrella. Each Task tracks the capability on a single surface, in its own repo, and is validated on its own. The parent can’t close until every sub is Done.

  • →PRs land against the level your change serves — link it with Part of #NNN. A surface Task for client work; the Server sub for the shared API/core. The parent stays an umbrella — it coordinates the subs, it doesn’t hold code of its own.
  • →Don’t invent parallel trackers. The acceptance criteria on the parent are the definition of done — tick those boxes, don’t write your own list.
  • →A Library Type parent carries one extra sub alongside its surfaces — a plugin-validation Task: at least one first-party metadata plugin must validate before the type counts as supported.
05

Definition of done

Code-complete is not done. A developer can mark code ready; only a maintainer testing on the real device moves a surface to Done — or sends it back.

To do → In progress → Needs validation → Done

Status is a Project field — one lane every Task moves through. A surface sits at To do until it’s picked up, moves to In progress while it’s built (the PR is opened and code-reviewed here), then to Needs validation when you’re ready — and you stop there; do not close it yourself. A maintainer runs it on the actual device, ticks the acceptance-criteria boxes, and moves it to Done.

A feature or library type only reaches Done when every one of its surfaces is Done. The same lane also runs the library-user acceptance suites (spec §4, run per library type on each surface), and the readiness-gate issues (spec §6 — first-run, upgrade safety, play-path security) gate the milestone right alongside them.

↩ If it fails on device, it loops back. The maintainer un-ticks the failing boxes, leaves notes, and moves the Task from Needs validation back to In progress until it’s fixed and re-submitted. There’s no separate “needs-changes” state: the bounce is just the status moving back, and the un-ticked boxes plus the maintainer’s comment carry the “why.”

Status is a field, so anyone with write access can move a card — that only a maintainer moves a card to Done is a rule this document sets, not a lock.

06

Readiness gates

Some bars a public release must clear belong to no single feature — first-run, upgrade safety, backup/restore, play-path security, observability (spec §6). Every feature being Done does not make the release shippable; the gates do.

A readiness gate is a Task carrying the readiness-gate label, in the v1.0 milestone. It rides the same status lane as everything else — but Needs validation here means a maintainer runs the real scenario: a fresh install, a real upgrade from the prior version, a wrong-profile stream URL that must be rejected.

Ship rule

Tag a release only when every Feature and Library Type parent is Done and every readiness-gate issue is Done. The gates sit in the milestone, and the release waits on both.

Contracts you must respect
08

The API contract

The client↔server API is one of three frozen contracts (Article 09). Client teams plan against it, so within a major it only ever grows.

Hard stop

Within /api/v1, changes are additive-only. Never rename or remove a response field, change a field’s type, or repurpose a status code on an existing endpoint. Break this and you break every shipped client.

  • ✓New functionality adds new fields or new endpoints — never mutates existing ones.
  • ✓Expose capability endpoints for feature detection. Clients ask what a server can do; they don’t sniff version numbers.
  • ×Removals go through the Deprecation / Sunset header flow only — never a silent drop. Contract tooling: issue #135.
09

Versioning

Semantic versioning, measured against three frozen contracts: the client↔server API, the plugin SDK, and the DB migration path. The bump is decided by what breaks — not by whether it felt like a feature.

BumpWhenExamples
MAJORBreaks a contract — a 1.x client can’t reach the server, a 1.x plugin won’t load, or a non-reversible migration.redesign client↔server API · plugin SDK v2 · drop an auth scheme
MINORAdds capability, backward-compatible — old clients and plugins keep working.ebooks/manga · Watch Party · new optional fields
PATCHFixes behavior, no new surface, no contract change.playback bug · metadata mismatch · crash fix

Design APIs additively to stay in minor-land (Article 08). Clients ride the server major — they float on minor/patch and check advertised capabilities. Plugins keep their own clock, gated only by the SDK major. Pre-release tagging on next is detailed in the release-engineering playbook.

10

Plugins & the trust tiers

A plugin’s tier is the scope of the promise behind it, not where its code happens to live. Three tiers; know which one you’re building for before you start.

First-party

Silo builds and owns it. silo.* id, Silo-Server org, default catalog, Silo release key.

tmdbtvdbmarkers-introdb
Community-vetted

Community-authored, maintainer-reviewed for function & safety only. community.* id, Silo-Community org, opt-in feed.

autoscan-arrrequests-seerr
External

Anyone, anywhere, installed by URL. No review — the server marks it Unverified.

Rule

If you build an acquisition or automation-stack adapter, it ships from Silo-Community under the community.* namespace — never Silo-Server, never silo.*. To enter the vetted tier it must be mechanism-only (a rescan trigger, a request hand-off); bundle an indexer, tracker, or source list and it’s rejected to External. Silo vouches that a vetted plugin works and is safe — never for what you point it at.

11

Opening a pull request

Every PR is one concern, traceable to a scope item, and green on the checks before a human looks at it.

Rule

Link the scope item with Part of #NNN. One concern per PR. Use a Conventional Commit subject (feat(playback): …). Disclose AI use in the body. PRs with no linked scope item will be questioned at review.

Rule

Before you request review, rebase your branch onto its base (main or next) and resolve conflicts locally, so the PR is current and reviews against the latest line — don’t pull the base in with merge commits. Rebasing your own branch is expected. The “never rebase a shared branch” rule (Article 01) still holds: it governs main, next, and any branch others have built on — not your unshared topic branch.

Integration

PRs land by squash-merge through the merge queue — one commit per PR on main/next, so history stays linear and reads as a changelog (your Conventional Commit title becomes that commit). A maintainer may rebase-merge a curated multi-commit PR where the internal history is genuinely atomic; merge commits are never used. Don’t base a branch on another un-merged branch — after the parent squashes, the child carries duplicate commits.
