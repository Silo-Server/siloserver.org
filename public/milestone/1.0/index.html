Silo v1.0

1Release summary

Silo v1.0 is a self-hosted media server for movies, series, and audiobooks, delivered with native clients on iPhone/iPad, Apple TV, Android phone, and Android TV, plus a web app for both viewing and server administration. The core promise is a complete browse → metadata → play experience at parity on every client, surrounded by a mature server: multi-profile households with access control, personalization, distributed and hardware-accelerated transcoding, casting, offline downloads, plugin extensibility, and migration tools for moving in from Plex, Jellyfin, and Emby — including drop-in compatibility for existing Jellyfin and Audiobookshelf clients.

2Platforms & clients

v1.0 ships four native clients plus a web app — five consumption surfaces — at feature parity for the core experience. The web app is also the server’s administration surface.

What “parity” means. Core-experience parity (browse → metadata → play, section 4) is mandatory and identical on every surface. Beyond that, parity is per-feature: each feature’s surface tags are its parity contract. Deliberate asymmetry — web-only admin, mobile-only downloads — is by design; a platform lacking a feature it isn’t tagged for is not a gap.

3Library types

Three library types ship in v1.0, each at full parity across every client. Each must clear the Core Experience bar in section 4.

Movies

TMDB metadata. Extras and edition labels where present.

Series

TMDB + TVDB metadata. Season/episode hierarchy, Next Up, specials, watched roll-up.

Audiobooks

Chapter navigation, playback speed, sleep timer, exact-second resume, author/narrator.

Sports content is supported as a series library via the Sportarr metadata plugin; there is no separate sports library type. Ebooks, manga, music, and podcasts are not v1.0 library types — see section 7.

Plugin contract. Library and marker support is delivered through plugins. Metadata: a library type counts as first-party-supported only once at least one first-party metadata plugin validates against it (Movies → TMDB, Series → TMDB + TVDB, Audiobooks → audiobook-metadata). Marker: Movie/Series libraries only. Request: the request system is first-party and extensible; community-maintained plugins can automate how requests are sourced. Autoscan: Silo owns the scan intake API; community-maintained adapters connect specific automation stacks or file-change sources.

4Core experience — Library-user acceptance

The Core Experience is the consumer’s experience, so it is written as library-user acceptance — the journey a person takes through any shipping library type, validated on every client — with a separate admin acceptance group for the server-side guarantees a user never touches directly. Budgets are p95 ranges on stated reference hardware. Each item states what is in and, where relevant, what is out (forward-only progress, single-quality transcode sessions, mid-stream track-switch restarts the transcode).

Library-user acceptance — every client (iOS / tvOS / Android phone / Android TV / web)
  • Find: the library appears in nav with an artwork grid/list, sort + filter, lazy-load, and a defined empty state; its items and people are searchable, results-as-you-type, filterable by type.
  • Understand: a detail screen shows full metadata, artwork, and related/cast where applicable.
  • Play: starts from detail and from resume; direct-play when codecs allow, remux/transcode fallback otherwise; HDR plays correctly on capable displays and tone-maps/transcodes to SDR elsewhere rather than failing. Cold-start transcode reaches first frame within a stated budget; seeking past the buffered region works; switching audio/subtitle mid-playback resumes at position (a brief rebuffer is expected). Adaptive bitrate is out of scope; manual quality selection is provided.
  • Keep my place: resume position is monotonic-forward across devices and profiles; two devices converge to the furthest position; mark watched/unwatched works and series roll-up updates; Continue Watching reflects it.
  • Recover: server unreachable mid-browse → non-destructive error + retry, no crash; token expiry → silent refresh, re-login only when the refresh token is dead; mid-stream drop → pause and auto-resume on reconnect. A no-metadata item still appears (filename title, placeholder art) and plays; a corrupt / 0-byte file gives a clear error without crashing the player or list; mixed-codec libraries work.
  • Accessibility: selected subtitles/captions render legibly and on time; core browse/detail honor the system text-size setting. Full screen-reader support and caption styling are post-1.0.
  • App lifecycle: backgrounding/resuming during browse and playback restores state without crash or forced re-login; rotate/resize does not crash the player.
  • In-player controls: play/pause/seek/scrub, audio + subtitle selection, next-item/autoplay where relevant.
TV-specific acceptance (tvOS / Android TV)
  • Every control in browse, detail, and player is reachable and operable by D-pad / remote alone; focus is always visible and never trapped.
  • Play/resume is reachable within a bounded number of D-pad moves from a focused tile; search uses the on-screen keyboard.
  • Mark watched / favorite is a focusable control on the detail screen, not a gesture.
Admin acceptance — server (validated by an admin, not a library user)
  • A configured folder scans to completion; rescans reflect added / changed / removed files, are idempotent (no dupes), and preserve existing watch progress & history.
  • Every item is matched to metadata; unmatched items stay visible and are manually fixable (identify, match-search, field + artwork edit).
  • Browse endpoints paginate; the first page of a large reference library (state the size, e.g. ~10k items) returns within a p95 budget (e.g. ≤1 s) while transcodes run.
  • Profile library access and content rating are enforced server-side on both browse and stream endpoints — the access resolver is the boundary, not UI hiding.
  • Release-gate criteria (first-run, upgrade safety, play-path security, observability) live in section 6.

Type-specific add-ons. Series: season/episode hierarchy, Next Up, specials, watched roll-up.   Audiobooks: chapter nav, playback speed, sleep timer, exact-second resume, author/narrator.   Movies: an edition label is shown where present; where a title has multiple versions, the server resolves a preferred version that plays by default.

Send to the TV app (SiloRemote)

iOStvOSAndroidAndroid TV

From the Silo mobile app, send a title to the Silo TV app on the same network and use the phone as a full remote. The TV app plays it directly from the server — the phone hands off and controls playback, it doesn’t relay or cast the video. Cross-platform, same-LAN.

  • From the Silo mobile app, send a title to a Silo TV app on the same LAN (Android phone → Apple TV, Android TV → Apple TV, and same-platform pairs); the TV app pulls the stream directly from the server, not from the phone.
  • After handoff the phone acts as a full remote (play/pause, seek, audio/subtitle, quality, volume) and stays in sync; it never transcodes or relays the stream.
  • Same-LAN only. If the TV app drops, the phone surfaces a disconnect and recovers rather than hanging.

  • On a host with a supported GPU, hardware encode/decode is auto-detected and used, with config to force or disable it; falls back to software cleanly.

Server, admin & operations

User management / invites / sharing

Web admin

Create accounts, invite people, and share specific libraries while keeping the rest private.

  • From web admin: create accounts, invite users, share specific libraries.
  • A shared user sees only granted libraries, enforced server-side. Not surfaced in the apps.

Admin impersonation

Web adminServer

An admin acts as a non-admin user to reproduce and diagnose what that user sees.

  • From web admin, an admin starts an impersonation session for a non-admin user and sees the experience as that user.
  • Impersonation cannot target another admin or escalate privileges; it ends cleanly and returns the admin to their own identity.

Requests

WebWeb admin

A self-contained request system: a user asks for a movie or show the server doesn’t have, an admin approves it, and when the content is later scanned in and identified it’s marked available and the requester is notified.

  • From the web app, a user searches for and requests a movie or show the server doesn’t have; a request for content that already exists is prevented, and if the title is already requested the user can add themselves to be notified when it becomes available instead of requesting it again.
  • A request moves through a status lifecycle (pending → approved/declined → available); an admin can approve or decline, and per-user request permissions/limits are enforced.
  • A user can be set to auto-approve on a per-user basis — their requests skip the pending state and go straight to approved.
  • The loop closes on its own: when matching content is scanned in and identified, the server marks the request available — no external service is required.
  • Everyone who requested or followed the title is notified when it becomes available.
  • The request system is extensible through a request-router plugin interface; community-maintained extensions can be installed to automate how requests are sourced.
  • Web only for v1: no request entry point ships in any first-party app; any built-but-hidden client path stays unreachable in release builds.

Device & override auditing

iOStvOSAndroidAndroid TVWebWeb adminServer

The server tracks every device a profile signs in from and resolves settings per device and per profile. Admins audit devices and overrides in web admin; users set overrides for their own devices and reorder their own profile’s sections.

  • The server registers each device a profile uses (id, name, platform, last-seen) and records each login as a session with device + IP; an admin lists devices in web admin — sorted by recency, with each device’s override count and the profiles using it — and opens a device to see its detail and overrides.
  • Settings resolve by precedence — device override → profile/user value → default — and the effective value reports its source (device / user / default).
  • A user sets and clears overrides for their own device (e.g. preferred quality, audio language, auto-skip intro/recap/credits, HDR, playback speed, subtitle sync) for the active profile; overrides are profile-scoped — switching profiles on the same device sees its own values.
  • An admin, from the device audit, can view and set both device-level overrides and profile-level edits (including a profile’s home section order) for any user, and remove a single override, all overrides for a key, or all overrides on a device.
  • Anomalies are surfaced — e.g. stale devices that still hold overrides, or outlier override counts — via saved views.
  • Boundary: device tracking is automatic from client-reported headers and enforced server-side; v1.0 has no device blocking/approval and no trusted/blocked device state.

Autoscan

Server

A self-contained scan intake system: an external source tells Silo that files changed, Silo maps the signal to a library/path, and the server performs a targeted rescan without waiting for the schedule.

  • Silo exposes an authenticated scan intake path that accepts a library/path change signal; when matching content is scanned in and identified, it appears in the library without waiting for the scheduled scan.
  • The loop closes on its own: the server owns signal intake, path mapping, debouncing, scan queueing, and result reconciliation — no external media-server refresh endpoint is required.
  • Signals are debounced so bursts of changes do not hammer the scanner.
  • The scan intake system is extensible through a source-adapter plugin interface; community-maintained extensions can be installed to connect automation tools, download clients, or other file-change sources.
  • Server only for v1: no first-party client UI ships for configuring source adapters; setup happens through server/admin configuration.

Push notifications (mobile)

iOSAndroidServer

Native push to phones and tablets for opted-in events, delivered through APNs and FCM via a first-party maintained relay. The relay only wakes the device; notification contents stay on the user’s Silo server and are fetched by the client after wake.

  • An opted-in event (e.g. a request was fulfilled) wakes iOS and Android through APNs/FCM; the push payload contains no media title, server URL, user name, or notification body.
  • A first-party maintained relay is used only to satisfy APNs/FCM delivery requirements; it receives an opaque device/registration target and never learns the server identity, notification contents, or library contents.
  • After wake, the client connects directly to its own Silo server to fetch pending notification contents and deep-link context; tapping the notification opens the relevant item once fetched.
  • A user registers/unregisters a device and chooses which event types push; tokens are revoked server-side on sign-out or session revoke.
  • tvOS and Android TV are out of scope, and no in-app notification inbox ships (post-1.0).

6Release-readiness gates

Cross-cutting criteria a public, self-hosted 1.0 must clear, independent of any single feature.

  • First-run: a fresh install reaches usable (create admin → add a library → scan → browse → play) via documented steps, with no manual DB or config-file editing on the happy path.
  • Upgrade safety: upgrading from the prior release runs migrations automatically and non-destructively; a failed migration aborts and leaves the DB on the prior schema; a documented rollback path exists.
  • Backup & restore: a documented backup/restore path covers the database, object storage, server config, and the secret-encryption key; restoring onto a clean host reproduces a working server. Redis is treated as disposable cache/session state, not a backup target.
  • Secret handling: provider tokens, SMTP credentials, S3 keys, watch-sync OAuth tokens, and plugin credentials are encrypted at rest, redacted in logs and API responses, and survive migration.
  • Observability: a failed scan / transcode / metadata match is visible to the admin (item identity + reason), not a silent no-op; /health and /ready reflect dependency outages.
  • Security of the play path: stream/media URLs require a valid, expiring, per-profile token; an unauthenticated or wrong-profile segment request is rejected; no shared/compiled-in secret grants playback or cast control across servers.
  • Frozen stability contracts: the client↔server API, plugin SDK, and DB migration path are versioned with a published min-version policy.
  • API versioning: stable public HTTP APIs are major-versioned by path (e.g. /api/v1); Silo 1.x releases preserve /api/v1 compatibility, additive changes remain in v1, and breaking changes require a new major API path.
  • Client/server compatibility checks: the server exposes a lightweight version/capabilities response, and first-party apps use it to check the API major and advertised capabilities before enabling features; apps are tested against the declared Silo 1.x server compatibility floor, unsupported API majors or below-minimum servers fail with a clear upgrade message, and features tied to missing same-major capabilities are hidden or degrade gracefully instead of forcing exact minor-version lockstep.
  • Published API reference: stable public integration surfaces — API keys, webhooks, scan intake, plugin SDK, and compatibility endpoints — are documented in versioned docs on siloserver.org; the self-hosted server does not serve public interactive API documentation by default.
  • Scope enforcement: a release build exposes no deferred surface — deferred routes are feature-flag-off or report capability unsupported, carry no client entry point, and are excluded from public docs; an automated release-build check enforces this (see section 7).
  • Provider compliance: first-party metadata/subtitle plugins comply with provider ToS and rate limits at public scale; bundled keys are audited.
  • Localization floor: dates/times display in the device locale and timezone; non-ASCII titles and paths scan, browse, and play correctly. Full UI translation is post-1.0.

7Not in v1.0

Explicitly held for later. “Deferred” is enforced, not aspirational: a feature is out of v1.0 only when no release build can reach or advertise it — the bar is reachability, not whether the code exists.

The gate tiers below apply to surfaces deferred across all release builds. A feature that is in-scope on one platform but absent on another is not a gating concern — that asymmetry is governed by each feature's surface tags (see section 2), enforced per-platform, not by these gates.

Release-gate policy for deferred surfaces
  • Default gate: no client entry point, removed from public docs, the capability probe reports unsupported, and the route stays authenticated — mounted but inert and unadvertised.
  • Strict gate: the above plus a release feature-flag off, so the route returns 404/501 in release builds. Used where a stray reachable surface is a reputational, legal, or maturity risk — e.g. ebooks/manga and Watch Party.
  • Removed: genuinely abandoned, no-roadmap, or misleading code is deleted rather than gated — e.g. client music shells.
  • Verified: a release-build check asserts deferred routes are flag-off / capability-unsupported / unlinked, so the boundary cannot silently drift (the section 6 scope-enforcement gate).
